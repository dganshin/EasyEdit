课程论文

基于模型编辑的大语言模型隐私知识清洗与保护研究

课程：智能系统综合实践

类型：研究型课程论文

关键词：大语言模型；模型编辑；隐私知识清洗；知识保持；过度拒答

# 摘要

大语言模型在预训练和后训练过程中可能形成对个人敏感信息的参数化记忆，并在特定提示下输出电话号码、邮箱等个人可识别信息（Personally Identifiable Information，PII）。仅依赖输出过滤或提示层拒答难以改变模型内部知识关联，而全量重训成本较高。针对这一问题，本文研究基于模型编辑的隐私知识清洗方法，目标是在降低目标隐私泄露的同时尽量保持同主体公开知识、同关系其他主体公开知识和通用知识。本文首先构建合成隐私评测基准，将隐私事实与公开事实分离，并通过低秩适配（Low-Rank Adaptation，LoRA）向 Qwen2.5-7B 注入可控隐私记忆，得到统一的合并后隐私泄露模型。随后，基于 EasyEdit 框架比较秩一模型编辑（Rank-One Model Editing，ROME）、Transformer 批量记忆编辑（Mass-Editing Memory in a Transformer，MEMIT）与隐私感知闭环编辑（Privacy-Aware Closed-loop Editing，PACE）的行为差异，并进一步设计副作用感知隐私编辑（Collateral-Aware Privacy Editing，CAPE）请求筛选策略，用于分析请求数量、覆盖主体范围和提示形式对过度拒答的影响。实验结果表明，MEMIT 在公开知识保持方面较稳定，ROME 呈现中间权衡区域，PACE 构成强隐私压制端点；加入副作用感知筛选后，极端公开知识崩塌得到缓解，验证了请求层约束对控制过度拒答具有作用。本文的主要贡献在于构建可复现的隐私知识清洗实验闭环，引入公开问题拒答率作为副作用指标，并系统揭示模型编辑隐私清洗中的隐私压制—知识保持权衡，为后续引入局部性约束和公开数据集基准提供基础。

关键词：大语言模型；模型编辑；隐私知识清洗；知识保持；过度拒答；ROME；MEMIT；PACE；CAPE

# 1  绪论

## 1.1  研究背景与意义

大语言模型已经成为智能系统的重要基础设施，被广泛应用于问答、办公自动化、代码生成和知识服务等场景。模型通过大规模语料学习语言规律和事实关联，在获得强生成能力的同时，也可能记忆训练语料或后训练数据中的具体事实。若这些事实涉及电话号码、邮箱、住址、工作单位等敏感信息，模型在特定提示下可能输出隐私内容，进而带来安全、合规和信任风险。

传统防护方式多发生在输出层，例如关键词过滤、安全分类器和拒答模板。这类方法部署便利，但没有直接改变模型内部已经形成的知识关联。对于改写询问、补全诱导、上下文诱导等攻击形式，外部过滤策略可能出现漏检。模型编辑则尝试在参数层面对特定事实关联进行局部修改，为隐私知识清洗提供了不同于重训和输出过滤的技术路径。

隐私清洗任务与普通事实修正不同。事实编辑通常希望把错误事实替换为正确事实，而隐私清洗希望模型对目标隐私查询不输出敏感值，同时保留同一人物的公开信息、其他人物的相关公开知识和一般知识能力。因此，该任务需要同时考察隐私压制、知识保持和过度拒答。只统计隐私查询上的拒答比例，容易把宽泛拒答误判为清洗效果。

从应用场景看，隐私清洗具有明确的现实需求。企业知识库、客服对话、邮件语料和个性化助手都可能包含半结构化个人信息。模型一旦在后训练阶段接触这些数据，后续即使删除原始文件，也可能仍在参数中保留可触发关联。对于部署方而言，理想策略应当在不重新训练整个模型的前提下，降低特定隐私值被输出的概率，同时尽量维持模型对非敏感问题的正常回答能力。

从研究方法看，隐私清洗需要将安全目标和效用目标放在同一实验框架中评估。若只报告隐私泄露下降，方法可能通过对整类人物问题拒答获得较好数值；若只报告公开知识保持，方法又可能没有真正压制敏感值。本文将二者合并为多目标问题，并把公开问题拒答率作为副作用指标，目的是让隐私保护效果和知识损伤同时可见。

## 1.2  国内外研究现状

训练数据记忆和隐私抽取研究表明，神经语言模型可能在特定条件下复现训练样本中的长片段或结构化信息[1-3]。电话号码、邮箱和身份标识等个人信息具有明显格式特征，既便于正则检测，也可能在补全式提示中被模型输出。机器遗忘和差分隐私从训练机制层面对隐私风险进行控制，但往往需要重新训练、访问原始训练数据或引入较高效用代价[4-5]。

模型编辑研究关注在不全量重训的情况下修改模型中的特定知识。ROME 通过定位事实关联对应的 MLP 层表征并执行局部更新来实现单条知识编辑[7]；MEMIT 将这种思路扩展到多事实批量编辑[8]；MEND、SERAC、IKE、GRACE 等方法则分别从元学习、外部记忆、上下文学习或适配器角度探索编辑机制[9-11]。EasyEdit 将多种编辑方法统一到同一工程框架中，为同模型、同数据、同评估协议下的横向比较提供了基础[14-15]。

现有模型编辑评测通常强调可靠性、泛化性和局部性，但隐私清洗还需要显式衡量公开知识是否被误伤。尤其在拒答式目标下，编辑器可能学到宽泛的拒答模式，使人物相关公开问题也被拒绝。本文将公开问题拒答率纳入评估，用于刻画隐私清洗中的副作用。

训练数据抽取研究通常从攻击者视角出发，衡量模型是否会在提示诱导下输出训练样本。机器遗忘研究则更关注从模型中移除某些样本或概念的影响[5,21-22]。与这两类工作相比，本文的实验对象更具体：在后训练注入的可控隐私事实上，观察标准模型编辑器是否能够实现目标隐私清洗，以及这种清洗会对公开事实造成怎样的副作用。

通用知识编辑 benchmark 多以 CounterFact、zsRE 或 LAMA 等事实性任务为核心，目标通常是把某个关系对象改为新对象，并通过 locality 指标观察非目标事实是否保持[16-17]。隐私清洗中的目标输出并非另一个公开事实，而是拒绝泄露敏感值；这使得 locality 的含义发生变化。公开事实保持率和公开问题拒答率因此成为本文实验中更贴合任务的局部性度量。

## 1.3  现有方法不足

面向隐私清洗的主要困难在于任务目标同时包含敏感值压制和非敏感知识保持。若编辑过弱，模型仍可能在补全或上下文提示下泄露隐私；若编辑过强，模型可能对同主体公开知识产生拒答，导致服务能力下降。该权衡关系使单一指标难以完整评价方法优劣。

另一个困难来自实验可控性。真实个人信息难以确认其是否存在于训练语料，也难以在公开论文中直接展示和复现。本文采用合成隐私评测基准，目的在于提供可控、可归因和可重复的实验环境。该设置适合机制分析，但外部有效性仍需要公开数据集和公开模型进一步验证。

第三个困难来自攻击提示的多样性。直接询问只覆盖最显式的泄露路径，改写询问、补全诱导、角色扮演和上下文诱导可能触发不同的内部关联。一个编辑请求若只约束 canonical prompt，可能无法覆盖续写式攻击；但如果对所有失败提示都反复再编辑，又可能导致拒答模式扩散。因此，请求构造本身也是隐私清洗效果的重要因素。

第四个困难来自评估指标的解释边界。自动 contains 匹配可以稳定统计目标值是否出现，但对自然语言回答质量的覆盖有限；正则指标可以识别电话和邮箱格式，但覆盖范围集中在结构化隐私类型；拒答标记能衡量安全倾向，但合理拒答和过度拒答还需要结合问题类型区分。本文在结果分析中保留这些边界，使实验结论与指标能力相匹配。

## 1.4  本文研究内容与贡献

本文围绕“基于模型编辑的大语言模型隐私知识清洗”构建完整实验流程。首先，构造包含隐私事实和公开事实的合成数据集；其次，通过 LoRA 注入形成统一的隐私泄露模型；再次，在同一模型和同一评估协议下比较 ROME、MEMIT、PACE 和 CAPE；最后，从隐私值包含率、公开知识包含式保持率和公开问题拒答率等维度分析各方法的行为边界。

本文贡献包括三点。第一，构建 private/public 解耦的隐私清洗评测基准，使目标隐私压制和公开知识保持能够同时度量。第二，基于 LoRA 构造可控泄露模型，并在 EasyEdit 框架下比较多种模型编辑策略。第三，提出 PACE 与 CAPE 两类 wrapper 层策略，用于分析残余泄露闭环再编辑和副作用感知请求筛选对过度拒答的影响。

在方法定位上，PACE 和 CAPE 均属于编辑器外层策略，而非新的底层参数更新算法。这样的设计使实验能够保持 ROME/MEMIT 原始实现相对干净，同时把研究重点放在隐私任务特有的请求选择和副作用控制上。该定位也便于后续将同一请求筛选思想迁移到其他编辑器或公开基准。

在结果解释上，本文以多端点对比替代单指标排序。MEMIT 展示高公开知识保持但隐私压制不足的端点，PACE 展示强隐私压制但过度拒答较高的端点，ROME 和 CAPE 则用于观察中间区域和请求筛选的影响。通过这种组织方式，论文能够更清楚地说明隐私清洗任务中的结构性权衡。

# 2  相关技术与问题定义

## 2.1  大语言模型中的参数化知识与隐私记忆

大语言模型可被视为在参数中编码了大量语言模式和事实关联。前馈层和注意力层共同参与知识存储与调用，其中部分研究指出 Transformer 前馈层具有类似键值记忆的作用。隐私记忆可以理解为模型在参数中形成的关于个人敏感事实的可触发关联。当查询提示与该关联匹配时，模型可能输出对应敏感值。

参数化知识的特点在于其并不以显式数据库记录形式存在，而是分布在大量权重和激活模式中。隐私事实一旦进入这种表示空间，删除原始训练样本并不必然消除模型输出该事实的能力。模型编辑的研究价值正来自这一点：它尝试在较低重训成本下修改局部知识关联，从而为后训练隐私清洗提供可操作路径。

## 2.2  LoRA 参数高效微调

LoRA 通过低秩矩阵近似对模型权重增量进行参数高效更新，在保持基础模型主体参数不变的情况下完成任务适配[6]。本文利用 MLP-only LoRA 将合成隐私事实与公开事实注入模型，使编辑前模型具有稳定、可测量的隐私泄露行为。该步骤用于构造统一实验起点，后续清洗效果由模型编辑阶段负责评估。

本文选择 LoRA 构造泄露模型，是因为它能够以较低成本向模型注入可控事实，而不需要重新训练全部参数。MLP-only 设置使注入更集中于事实表达相关模块，也便于后续使用 ROME/MEMIT 等编辑器观察 MLP 层编辑效果。该设计将“构造泄露对象”和“执行隐私清洗”分离，减少实验解释中的混杂因素。

## 2.3  模型编辑与 EasyEdit 框架

模型编辑旨在以较低代价修改模型对特定知识的响应。EasyEdit 提供统一接口，使 ROME、MEMIT 等方法可以在相同输入格式和评估流程下运行。本文将 EasyEdit 作为基础工程框架，但不修改 ROME 和 MEMIT 的底层更新公式，所有改进集中在数据构造、请求选择和评估分析层面。

统一框架对本文尤为重要。若不同方法使用不同数据预处理、不同生成参数或不同评估脚本，结果差异可能来自工程细节而非方法本身。本文将请求格式、生成设置、full private/public 评估和结果汇总统一在同一流水线中，使 ROME、MEMIT、PACE 和 CAPE 的比较具有更清晰的控制变量。

## 2.4  ROME 与 MEMIT 方法概述

ROME 面向单条事实编辑，通过定位事实关联在模型中的关键层并执行秩一更新来改变目标输出[7]。MEMIT 面向批量事实编辑，在多个层上估计更新并同时注入多条编辑约束[8]。两者均可用于隐私拒答式编辑，但其原始设计目标并非隐私清洗，因此需要结合公开知识保持和过度拒答指标重新评价。

ROME 和 MEMIT 的差异为本文提供了两个重要观察角度。ROME 更适合单条或少量事实编辑，可能在局部压制上更直接；MEMIT 面向批量编辑，可能对整体模型行为扰动较小。将二者放在隐私拒答目标下比较，可以观察标准编辑器在非替换式目标上的适应性。

## 2.5  隐私知识清洗任务定义

设原模型为 M，编辑后模型为 M'，待清洗隐私事实集合为 K_F，需要保留的公开事实集合为 K_R。隐私知识清洗要求 M' 在隐私查询集合上降低目标敏感值输出，在公开查询集合上保持正确公开事实，并避免把非敏感问题泛化为拒答。

在本文设置中，K_F 包含需要清洗的电话号码和邮箱等隐私事实，K_R 包含职业、学校、雇主和家乡等公开事实。查询集合 Q_F 覆盖多类攻击提示，查询集合 Q_R 覆盖同主体公开知识、同关系其他主体公开知识和通用知识。这样的划分使隐私清洗被定义为一个多目标优化问题，而非单一拒答任务。

$$\mathcal{J}(M') = \operatorname{Leakage}(M', \mathcal{K}_{F}) + \lambda \operatorname{UtilityLoss}(M, M', \mathcal{K}_{R}) + \mu \operatorname{OverRefusal}(M', \mathcal{K}_{R})$$

## 2.6  评价指标与审计口径

本文将隐私值包含率作为目标隐私泄露的主要自动指标。该指标判断目标敏感值是否作为归一化子串出现在模型输出中，适合衡量电话号码、邮箱等显式隐私值泄露。PII 格式匹配率进一步统计输出是否出现目标格式的敏感串，但仅覆盖电话和邮箱等可正则检测类型。

公开知识包含式保持率衡量公开事实答案是否出现在公开问题输出中。由于该指标采用包含式匹配，它反映的是目标公开值是否被提及，而非完整语义正确率。公开问题拒答率统计公开问题中出现拒答标记的比例，用于衡量过度拒答副作用。

三类指标之间存在天然张力。隐私值包含率越低，说明目标敏感值越少出现在输出中；公开知识包含式保持率越高，说明非敏感事实越可能被保留；公开问题拒答率越高，则说明拒答行为可能扩散到保留集合。本文在表格和图中同时呈现这三类指标，以避免将强拒答误读为高质量清洗。

需要强调的是，本文的自动指标服务于可复现实验比较。对于最终部署场景，仍需结合人工标注、语义相似度或安全评测模型进一步判断回答质量。例如，模型可能没有输出目标值，但生成了含糊或错误的公开回答；也可能没有命中拒答标记，却以礼貌回避方式拒绝回答。自动评估提供稳定基线，人工评估可作为后续补充。

$$\operatorname{PVC}=\frac{1}{|\mathcal{Q}_{F}|}\sum_{q \in \mathcal{Q}_{F}}\mathbf{1}\left[\operatorname{norm}(v_q)\subset\operatorname{norm}(M'(q))\right]$$

$$\operatorname{PC}=\frac{1}{|\mathcal{Q}_{R}|}\sum_{q \in \mathcal{Q}_{R}}\mathbf{1}\left[\operatorname{norm}(a_q)\subset\operatorname{norm}(M'(q))\right]$$

$$\operatorname{PRef}=\frac{1}{|\mathcal{Q}_{R}|}\sum_{q \in \mathcal{Q}_{R}}\mathbf{1}\left[M'(q)\in\mathcal{S}\right]$$

# 3  隐私知识清洗框架设计

## 3.1  总体框架

本文框架由四个阶段组成：合成隐私数据构造、可控泄露模型构造、模型编辑执行和多维评估分析。数据阶段明确区分隐私事实和公开事实；注入阶段通过 LoRA 构造统一的待编辑模型；编辑阶段比较基础编辑器与闭环策略；评估阶段同时报告隐私压制、知识保持和过度拒答。

总体框架的设计原则是将实验对象、编辑算法和评估协议解耦。实验对象由合成数据和 LoRA 注入决定，编辑算法由 EasyEdit 中的 ROME/MEMIT 承担，请求选择由 PACE/CAPE 控制，评估协议由统一生成和统计脚本实现。这种分层设计能够清楚地区分“模型是否已经学会隐私事实”“编辑器是否压制隐私输出”“请求选择是否引入副作用”等问题。

## 3.2  合成隐私评测基准构建

评测基准包含 100 个虚拟人物。每个人物具有电话号码、邮箱等隐私事实，以及职业、学校、雇主、家乡等公开事实。公开事实进一步划分为同主体公开知识、同关系其他主体公开知识和通用知识，用于观察编辑是否只影响目标隐私，还是扩散到相邻公开知识。

合成数据的优势在于 ground truth 完全可控。每个隐私值和公开值均由实验脚本生成，因此可以精确判断模型输出是否包含目标值。与真实 PII 相比，合成数据避免了公开论文中展示真实个人信息的伦理和合规问题，同时保留了电话号码、邮箱等结构化隐私格式，使泄露检测具有可重复性。

该基准还特别设计了多种攻击提示。直接询问用于衡量最显式的泄露风险；改写询问用于观察语义等价表达是否绕过编辑；补全诱导用于测试模型在续写场景下是否延续个人资料；角色扮演和上下文诱导用于模拟更复杂的交互式提示。这些攻击类型共同构成隐私清洗的压力测试集合。

## 3.3  可控隐私泄露模型构造

本文使用 MLP-only LoRA 将隐私事实和公开事实共同注入 Qwen2.5-7B，并将适配器合并为待编辑模型。合并后模型在隐私查询上具有较高泄露率，同时在公开问题上保持较高包含式保持率，说明该模型适合作为统一编辑起点。

选择合并后的模型作为编辑对象还有一个重要原因：ROME 和 MEMIT 在运行时直接修改模型参数，如果继续依赖外部 adapter，会增加编辑对象和评估路径的不确定性。将 LoRA adapter 合并后，所有方法面对的是同一个普通 causal language model，便于比较编辑前后的参数级影响。

## 3.4  基于 ROME/MEMIT 的隐私拒答编辑

第一轮编辑将隐私事实对应请求改写为拒答目标，使模型在目标隐私查询上倾向于拒绝输出敏感值。ROME 和 MEMIT 使用各自标准超参数运行。该阶段用于观察基础编辑器在隐私清洗任务中的默认行为。

拒答目标的设置反映了隐私清洗的任务特点。对于电话号码或邮箱，目标不是把旧值替换为另一个值，而是让模型避免输出该敏感值。因此编辑请求将原始隐私查询映射到安全拒答表达。该设计能够直接评估编辑器是否可以把事实编辑机制迁移到隐私保护目标。

## 3.5  PACE 隐私感知闭环编辑

PACE 是仅基于残余泄露风险的闭环再编辑策略，可视为无副作用约束的基础请求选择策略。其核心思想是在第一轮编辑后重新评估残余泄露，从仍然泄露的样本中构造第二轮编辑请求，再执行补充编辑。该策略保持底层编辑器公式不变，通过残余泄露驱动请求构造。形式上，PACE 可以多轮迭代，直到残余泄露低于阈值或达到编辑预算；本文实验采用二轮实例化，是为了控制运行成本和过度拒答风险。

闭环策略的动机来自基础编辑器的不完全覆盖。单轮 direct-only 请求主要约束 canonical query，面对改写、补全或上下文提示时可能仍有残余泄露。PACE 将评估结果反馈到请求构造阶段，使失败样本能够被再次编辑。该设计提高了隐私压制强度，也带来更高的副作用风险。

算法 1  隐私知识清洗总体流程

输入：基础模型 M，合成隐私数据集 D，基础编辑器 E，评估提示集合 Q。

输出：编辑后模型 M'，评估报告 R。

1. 构造隐私事实集合 K_F 与公开事实集合 K_R；

2. 通过 LoRA 将 K_F 与 K_R 注入模型，得到 M_leak；

3. 根据 K_F 构造第一轮编辑请求集合 R_1；

4. 使用编辑器 E 对 M_leak 执行第一轮编辑，得到 M_1；

5. 在攻击提示集合上评估 M_1，得到残余泄露集合 C；

6. 根据 PACE 或 CAPE 策略构造第二轮请求集合 R_2；

7. 基于 R_1 与 R_2 得到最终编辑模型 M'；

8. 计算隐私泄露、公开知识保持和过度拒答指标。

## 3.6  CAPE 副作用感知请求筛选

CAPE 是引入公开知识锚点、主体预算和请求数量控制的副作用感知请求筛选策略。该策略从残余泄露候选中选择请求，同时加入公开知识锚点约束、每主体编辑预算和总请求数限制。其目标是在保留残余泄露修复能力的同时降低过度拒答风险。

CAPE 的核心假设是，并非所有残余泄露候选都适合作为第二轮编辑请求。覆盖主体过多、提示形式过杂或与公开知识高度耦合的请求，可能把拒答模式扩散到保留集合。通过限制每个主体的编辑预算、设置总请求上限并优先选择风险更明确的候选，CAPE 尝试把闭环再编辑从“尽可能多地压制”调整为“在副作用可控范围内压制”。

$$\operatorname{score}(q)=\operatorname{Risk}_{\mathrm{leak}}(q)-\lambda \operatorname{Risk}_{\mathrm{loc}}(q)$$

算法 2  CAPE 副作用感知请求筛选

输入：残余泄露候选集合 C，公开知识评估结果 P，每主体预算 B，总请求上限 T。

输出：第二轮编辑请求集合 R_2。

1. 对每个候选 q 计算泄露风险和局部性风险；

2. 若候选所属主体的公开知识保持低于阈值，则跳过该主体；

3. 按隐私值包含、PII 格式匹配和敏感格式输出优先级排序；

4. 在不超过每主体预算 B 的前提下选择候选；

5. 当总请求数达到 T 时停止；

6. 将选中候选规范化为第二轮拒答编辑请求。

# 4  实验设置与结果分析

## 4.1  实验设置

实验使用 Qwen2.5-7B 作为基础模型，编辑对象为合并后的隐私泄露模型。所有方法使用相同合成隐私数据集、相同第一轮请求和相同 full private/public 评估集合。MEMIT 采用其标准 MOM2 配置；ROME 使用对应的 Qwen2.5-7B 超参数。

实验控制变量主要体现在三个方面。第一，所有编辑方法均以同一个合并后泄露模型为起点，避免模型版本差异影响结论。第二，隐私评估和公开评估使用同一批 prompt 集合，避免不同方法面对不同难度的查询。第三，结果汇总采用同一套指标脚本，使指标口径在横向比较中保持一致。

由于 MEMIT 的标准配置需要 MOM2 statistics，本文按照其方法要求为被编辑模型生成对应统计文件。ROME 配置中不启用 MOM2，因此二者在辅助统计依赖上存在方法自身差异。本文保留这种差异，将其视为各方法标准运行条件的一部分。

## 4.2  待编辑隐私泄露模型验证

编辑前模型的隐私值包含率为 0.9387，公开知识包含式保持率为 0.9766，说明 LoRA 注入同时形成了稳定隐私记忆和公开事实记忆。该结果保证后续编辑实验具有明确起点：若编辑后隐私泄露下降，可归因于模型编辑；若公开知识保持下降，则可视为编辑副作用。

敏感格式输出率在编辑前达到 0.9927，说明模型不仅能输出目标隐私值，也倾向于在相关提示下生成电话或邮箱格式内容。该现象为后续区分“目标值泄露”和“敏感格式幻觉”提供了基础。一个方法即使降低了目标值包含率，仍可能保留敏感格式输出倾向，因此本文同时报告这两个维度。

公开知识包含式保持率接近 1，说明公开事实在注入模型中同样被稳定记忆。若编辑后公开知识保持率大幅下降，更合理的解释是编辑过程造成了保留损伤，而非模型起点缺乏公开知识回答能力。这一点对于评价 PACE 和 CAPE 的副作用尤其重要。

图 3 将各方法放入隐私压制和知识保持二维空间中，可以更直观地看到方法端点。理想方法应位于右上角，即公开知识保持率高且隐私值包含率低。当前方法均未达到这一位置。图 4 进一步说明，高隐私压制方法往往伴随更高公开问题拒答率，过度拒答是当前隐私清洗中的主要副作用来源。

案例分析进一步说明，隐私清洗评价需要同时观察拒答对象和保留对象。若模型对公开职业、学校或雇主信息也拒绝回答，则实际系统可用性会下降。较好的清洗策略应当在目标隐私值上拒绝输出，同时保留公开事实回答能力。

典型案例还显示，敏感格式输出与目标值泄露并不完全一致。模型可能没有输出真实目标电话号码，却生成另一个电话号码格式字符串；这种行为不构成目标值泄露，但在隐私安全场景中仍然值得关注。因此，本文同时报告隐私值包含率、PII 格式匹配率和敏感格式输出率，以区分不同层次的风险。

合成基准适合可控机制分析，但公开基准有助于验证结论外推性。后续实验应区分真实 PII 清洗、虚构人物遗忘和通用事实编辑三类任务，避免把不同任务的指标直接混用。本文当前结果来自可控隐私注入场景，公开数据集扩展将用于检验过度拒答和隐私压制权衡是否稳定存在。

Enron 邮件语料适合构造真实邮件场景下的 PII 检测和清洗任务[20]；The Pile 与 Pythia 可用于分析公开预训练数据和模型规模对记忆风险的影响[18-19]；DEPN setting 与隐私神经元编辑相关，可作为参数定位类方法对照[23]；TOFU 属于虚构遗忘基准，适合考察忘记目标人物与保留非目标知识之间的关系[22]；CounterFact、zsRE 和 LAMA 更偏通用知识编辑，可用于补充 locality 对照[16-17]。

公开扩展实验还需要明确评价层次。Enron 更适合测试真实邮件中的结构化隐私值清洗；TOFU 更适合测试虚构人物知识遗忘；CounterFact、zsRE 和 LAMA 更适合测试普通事实编辑的局部性。三类基准回答的问题不同，因此后续结果应分别报告，不宜合并为单一排名。本文提出的公开问题拒答率和隐私压制—知识保持分析可以作为跨基准比较的共同视角。

## 4.10  有效性与局限性分析

本文实验的内部有效性来自统一模型、统一数据和统一评估协议。主要局限在于合成人物规模有限，公开事实类型仍较少，自动 contains 指标只能近似衡量目标值是否出现，尚需人工语义评估补充。真实预训练隐私泄露还涉及数据来源追溯、合规边界和跨语境攻击，需在公开基准上进一步验证。

外部有效性方面，合成隐私数据能够提供清晰 ground truth，但真实世界中的个人信息来源更复杂，可能与公共事实、上下文身份和多轮交互深度耦合。本文结果能够说明模型编辑隐私清洗中的机制性风险，尤其是过度拒答和请求选择副作用，但真实部署仍需要在更复杂数据和更严格人工评估下验证。

构念有效性方面，公开知识包含式保持率对回答质量的刻画仍然有限，公开问题拒答率也依赖拒答标记集合。后续可以引入语义匹配模型、人类评估和安全分类器，对回答正确性、拒答合理性和敏感性进行更细粒度标注。这样可以把本文的自动评估框架扩展为更完整的多层评估体系。

统计稳定性方面，当前实验使用 100 人规模的合成数据集，能够观察主要趋势，但更细粒度的类别差异仍需要更大规模样本支持。例如，不同隐私类型、不同公开关系、不同攻击模板之间可能存在交互影响。后续可通过扩大人物数量、增加关系类型和引入多随机种子，进一步评估结果方差。

# 5  系统设计与软件实现

## 5.1  系统模块划分

系统包含数据构造、LoRA 注入、编辑请求构造、模型编辑运行、隐私生成评估、公开知识评估和结果汇总模块。各模块通过 JSON、JSONL 和 Markdown artifact 连接，便于复查输入输出。

这种模块化设计使实验流程具有可替换性。若后续加入新的编辑器，只需适配请求输入和输出评估文件；若加入新的数据集，只需保持隐私事实、公开事实和评估提示的字段语义一致。系统实现因此既服务于当前实验，也为后续 baseline 扩展提供接口。

## 5.2  实验流水线与 artifacts 管理

每轮实验保存请求文件、case id、评估结果、摘要文件和运行日志。服务器负责 GPU 编辑与生成，本地负责脚本维护、结果分析和论文生成。

artifact 管理的核心原则是保留小型、可审计、可复现的结果文件，而不把模型权重和大规模缓存纳入论文仓库。请求文件记录编辑目标，评估 JSON 记录逐样本结果，summary 文件记录主要指标，运行日志记录 pipeline 状态。通过这些文件，可以在不重新运行 GPU 实验的情况下复核论文表格。

## 5.3  本地与服务器协同机制

本地 Windows 仓库用于代码和文档管理，AutoDL Linux 服务器用于运行大模型实验。同步方式采用 git pull/push，避免在服务器上进行大规模手改。

本地与服务器分工还降低了实验过程中的风险。本地环境适合进行脚本修改、论文生成和结果分析；服务器环境适合加载大模型、执行编辑和批量生成。通过 git 同步代码，通过 artifacts 同步小型结果，可以避免在服务器上临时修改导致版本不可追踪。

## 5.4  主要脚本与输入输出

主要脚本包括隐私数据生成、LoRA 训练数据构造、ROME/MEMIT 编辑、PACE/CAPE 请求构造、隐私泄露评估、公开知识评估和汇总分析脚本。每个脚本均以显式路径输入输出，减少实验复现中的隐式状态。

输入输出显式化对于模型编辑实验尤其重要。模型路径、数据集路径、请求路径、输出目录和运行标签均需要在命令中明确给出。这样可以避免不同实验结果覆盖，也能确保论文中每个指标都能追溯到对应 artifact。

## 5.5  复现注意事项

MEMIT 需要 MOM2 statistics，且应针对实际被编辑的合并模型生成。公开数据集缓存、模型权重和统计文件体积较大，不应作为论文 artifact 提交；论文只保留小型结果文件、摘要和图表。

复现过程中还需要注意 GPU 显存和数据缓存。MEMIT 统计文件生成会消耗较长时间，但生成后可复用；Wikipedia 等外部数据应在无 GPU 环境下预取，避免显卡租用期间等待下载。实验开始前应执行最小 preflight 检查，包括模型存在性、数据集规模、请求数量和输出目录是否会覆盖旧结果。

# 6  总结与展望

本文围绕基于模型编辑的大语言模型隐私知识清洗问题，构建了从合成隐私评测基准、LoRA 可控隐私注入、模型编辑执行到副作用审计的完整实验闭环。该流程使待清洗隐私事实、需要保留的公开事实以及多类攻击提示均具有明确 ground truth，从而能够在统一协议下比较不同编辑策略。

实验表明，不同编辑策略在隐私压制和知识保持两个目标上呈现差异化端点。MEMIT 表现出较强的公开知识保持能力，但在拒答式隐私清洗目标下压制强度有限；ROME 在隐私值输出压制和公开知识保持之间形成中间区域；PACE 作为仅基于残余泄露风险的闭环再编辑策略，构成强隐私压制端点，并为分析请求预算带来的副作用提供必要对照。

本文进一步引入公开问题拒答率作为副作用指标，用于区分合理隐私拒答和公开知识误伤。CAPE 在 PACE 的闭环框架中加入公开知识锚点、主体预算和请求数量控制。实验结果显示，副作用感知请求筛选能够缓解部分极端过度拒答现象，并验证请求数量、覆盖主体范围和提示形式会影响隐私清洗副作用。

从方法启示看，隐私知识清洗应被视为多目标约束问题。高质量清洗策略需要同时建模目标隐私、公开知识锚点和攻击提示类型，而不仅追求隐私值包含率最低。后续可以将 CAPE 从启发式请求筛选扩展为显式约束优化，使泄露风险、局部性风险和攻击覆盖率共同进入请求评分函数。

从实验扩展看，本文当前结果来自 Qwen2.5-7B 合成隐私 V2 设置。后续将优先引入 TOFU、Enron small masked subset、CounterFact 或 zsRE 等公开基准，并补充 GPT-J-6B、Pythia-6.9B 或 LLaMA-2-7B 等公开模型，以检验合成设置中的隐私压制—知识保持权衡是否能够外推到更通用的数据和模型环境。

从系统应用看，模型编辑可以与检索增强、输出过滤和访问控制共同组成分层隐私防护体系。模型编辑适合降低参数化记忆中的目标关联，输出过滤适合处理生成阶段的残余风险，访问控制适合限制高风险查询场景。本文的实验和分析为参数级隐私清洗环节提供了可复现证据，也为后续公开基准实验和局部性约束方法设计提供了基础。
